## 2. Python Standards

### 2.1 Version and Style
- Python **3.11+**
- Type hints on all function signatures (mandatory)
- Docstrings on all public methods and classes (Google style)
- Line length: 100 characters maximum
- Formatter: `black` (run before every commit)
- Linter: `ruff`

### 2.2 Naming Conventions

| Entity | Convention | Example |
|--------|-----------|---------|
| Module | `snake_case` | `fred_source.py` |
| Class | `PascalCase` | `FREDSource`, `SignalMetrics` |
| Function / method | `snake_case` | `compute_carry()` |
| Variable | `snake_case` | `signal_series` |
| Constant | `UPPER_SNAKE` | `DEFAULT_LOOKBACK = 252` |
| Config key | `snake_case` | `fast_window: 50` |
| DataFrame column | `snake_case` | `forward_return`, `is_outlier` |
| Signal identifier | `{asset}_{type}` | `fx_carry`, `rates_trend` |

### 2.3 Imports

Always in this order, separated by blank lines:
```python
# 1. Standard library
import os
from datetime import date
from typing import Dict, List, Optional

# 2. Third-party
import pandas as pd
import numpy as np

# 3. Internal — always use absolute imports from src/
from src.data.store import DataStore
from src.signals.base import Signal
```

Never use relative imports (e.g., `from .base import Signal`).

## 3. Data Conventions

### 3.1 DatetimeIndex Standard
- **Always UTC** — convert on ingest, never after
- **Business days only** — no weekends, no market holidays for daily data
- **No timezone-naive indices** in any DataFrame that persists to disk

```python
# Correct
df.index = pd.to_datetime(df.index, utc=True)

# Wrong — timezone-naive
df.index = pd.to_datetime(df.index)
```

### 3.2 Return Calculation
- Use **log returns** for all statistical calculations (IC, Sharpe, etc.)
- Use **simple returns** for compounded P&L and final performance reporting
- Always be explicit — never mix the two in the same calculation

```python
# Log returns — for statistics
log_ret = np.log(prices / prices.shift(1))

# Simple returns — for P&L
simple_ret = prices.pct_change()
```

### 3.3 Signal Normalisation
- All signals passed to the portfolio engine must be normalised to [-1, 1]
- Default method: rolling z-score clipped at ±3, then divided by 3
- Cross-sectional signals: use rank normalisation within universe at each date

### 3.4 Forward Return Calculation
- Always use **close-to-close** at specified horizon
- Always shift by +1 period to avoid lookahead (signal at t uses return t+1 to t+1+horizon)
- Label clearly: `fwd_ret_1d`, `fwd_ret_5d`, `fwd_ret_21d`

```python
# Correct — no lookahead
fwd_returns = log_returns.shift(-horizon - 1)  # +1 for execution lag

# Wrong — looks into the future
fwd_returns = log_returns.shift(-horizon)
```

### 3.5 NaN Policy
- **Into signal engine:** no NaN values (DataCleaner guarantees)
- **Signal output:** NaN permitted only at the start (warm-up period)
- **Into portfolio engine:** no NaN values in combined signal
- Explicit NaN handling required — never let pandas silently propagate NaN

### 3.6 Data contracts by layer (5.7+ / DD-010)

Different layers consume data in different shapes by design:
- **Signal layer:** `Dict[catalogue_variable_name, pd.Series]`. Each
  Series is a 1-D time series. This is the 5.7 contract.
- **Portfolio layer (sizer, constructor):** wide DataFrame panel
  (rows = dates, columns = instruments). Used for cross-sectional vol
  math and risk parity.
- **Backtest engine:** consumes the signal-layer contract on its
  public API; translates to the portfolio-layer panel at one explicit
  boundary (`engine._assemble_price_panel`). Reads instruments from
  `signal.instruments` (DD-010); does NOT accept
  `portfolio_config['instruments']`.

When adding new components, prefer the signal-layer contract by
default. The portfolio panel is a deliberate exception, retained
because cross-sectional operations are naturally expressed there. See
PROGRESS.md §5.7 "option A hybrid" for the rationale.

### 3.7 Frequency-mismatch policy (DD-004 + DD-010 follow-up)

Variables have native frequencies (FRED interbank rates are monthly,
FRED `DFF` is daily, Yahoo daily closes are daily). When a consumer
requests a finer frequency than the variable's native cadence:

- **The catalogue forward-fills** from the most recent published value
  on each business day in the request range. This is the DD-004
  honest-frequency policy: the value on day `t` is the most recently
  published value as of `t`. No future values are used.
- **Never interpolate.** Linear, spline, or model-based interpolation
  would introduce lookahead bias. Forbidden.
- **Runners and signal compute() consume daily everywhere.** Do not
  fetch compute inputs at a coarser frequency to "match the signal's
  natural frequency" — the resulting signal output would have
  coarse-grained dates that won't join with daily prices. The evaluator's
  frequency layer resamples signal and prices together to the signal's
  natural evaluation frequency.

## 4. Configuration Conventions

### 4.1 Loading Configs
```python
import yaml
from pathlib import Path

def load_config(name: str) -> dict:
    path = Path("configs") / name
    with open(path) as f:
        return yaml.safe_load(f)
```

### 4.2 YAML Format
```yaml
# configs/signals/fx_carry.yaml
signal:
  name: fx_carry
  asset_class: fx
  signal_type: carry
  frequency: daily

parameters:
  fast_window: 1
  n_long: 3
  n_short: 3
  smooth_days: 1

data_requirements:
  - DFF         # Fed Funds Rate
  - EURIBOR3M
  - GBPRATE3M

known_limitations:
  - "Rate differential proxies forward premium — approximation only"
  - "Does not include actual FX rollover costs"
```

### 4.3 Environment Variables
Never hardcode credentials. Always load from environment:

```python
from dotenv import load_dotenv
import os

load_dotenv()

FRED_API_KEY = os.getenv("FRED_API_KEY")
if not FRED_API_KEY:
    raise EnvironmentError("FRED_API_KEY not set. Check .env file.")
```

`.env` file format:
```
FRED_API_KEY=your_key_here
ANTHROPIC_API_KEY=your_key_here
IB_HOST=127.0.0.1
IB_PORT=7497
IB_CLIENT_ID=1
QUANDL_API_KEY=your_key_here
```

## 5. Error Handling

### 5.1 Custom Exceptions
All custom exceptions defined in `src/exceptions.py`:

```python
class TradingSystemError(Exception): pass
class DataFetchError(TradingSystemError): pass
class DataGapError(TradingSystemError): pass
class DataValidationError(TradingSystemError): pass
class LookaheadError(TradingSystemError): pass
class SignalComputationError(TradingSystemError): pass
class InsufficientDataError(TradingSystemError): pass
```

### 5.2 Rule
- Never catch generic `Exception` without re-raising or logging
- All errors must surface — no silent failures
- Use `loguru` for structured logging at appropriate levels

```python
from loguru import logger

# Correct
try:
    df = source.fetch(ticker, start, end)
except DataFetchError as e:
    logger.error(f"Failed to fetch {ticker}: {e}")
    raise

# Wrong
try:
    df = source.fetch(ticker, start, end)
except:
    pass  # Never do this
```

## 8. Reporting and Output Conventions

Added in Milestone 5.6. All code that produces report artifacts (plots,
results files, manifests) must follow these conventions.

### 8.1 Use `OutputManager` for all outputs

Never write directly to `reports/`. Always go through `OutputManager`:

```python
from src.reporting.output_manager import OutputManager

mgr = OutputManager()  # defaults to ./reports
run = mgr.new_variable(
    name="fx_carry",
    config={"frequency": "monthly", "start": "2010-01-01"},
)
# run.path → reports/variables/2026-05-15_14-30-22_fx_carry/
# run.plots_dir → run.path / "plots"
(run.path / "results.md").write_text(report)
run.finalize()  # writes manifest + appends to index
```

The manager guarantees: timestamped folder, manifest capturing git state,
appropriate index update, and run-kind isolation.

### 8.2 Choose the right run kind

Three run kinds, three separate folders. Pick based on the *purpose* of the
output, not what kind of code is producing it.

| Kind | Use for | Index tracked? |
|---|---|---|
| `exploratory` | Notebook research, one-off experiments, hypothesis tests | No |
| `variable` | Formal variable evaluations (signal IC tables, etc.) | Yes |
| `strategy` | Production strategy backtests | Yes |

A notebook can absolutely produce `variable` outputs if the work is formal
enough to be referenced later. Conversely, a script can write to `exploratory`
if it's truly a one-off.

### 8.3 Plots: use the shared library

All plotting goes through `src/reporting/plots.py`. Five reusable functions:
`plot_cumulative_returns`, `plot_ic_over_time`, `plot_drawdown`,
`plot_signal_heatmap`, `plot_correlation_matrix`.

Default to PNG at 150 DPI. SVG available via `save_format="svg"` when sharper
output is needed (e.g. for inclusion in a write-up).

Do not write one-off matplotlib boilerplate inside notebooks or scripts. If
a useful plot doesn't exist in the library yet, add it there and write a test.

### 8.4 Notebooks

Notebooks live in `notebooks/` and are for exploration only. See
`notebooks/README.md` for the full lifecycle rules. Critical ones:

- **No production logic in notebooks.** If code is reusable, move to `src/`.
- **Never import from another notebook.** That's a signal to refactor to `src/`.
- **Always write outputs via `OutputManager.new_exploratory()`.** Don't dump
  plots into the notebook's own directory.
- **Naming**: `YYYY-MM-DD_short_description.ipynb`. Date is when started.

### 8.5 Manifests

Every output run writes a `manifest.json` with:
- Git commit SHA (or `"unknown"` if not in a repo)
- Git dirty flag (warns if uncommitted changes affected the run)
- ISO 8601 UTC timestamp
- Python version + platform string
- `config` dict — caller-provided run parameters
- `extra` dict — anything else worth recording

When debugging an old result, the manifest tells you exactly what to check
out and re-run to reproduce. If `git_dirty` is true, full reproduction is
not guaranteed even from the commit alone.

### 8.6 Indexes

Each run kind has its own `reports/{kind}/index.csv` updated on every run.
`reports/variables/index.csv` and `reports/strategies/index.csv` are git-tracked
so the formal run history is visible without checking the data directory.
`reports/exploratory/index.csv` is gitignored — it's your scratch space.

Use the index when you need to find a past run by date, run_id, or commit.

## 10. Handoff Verification Protocol

Added in 2026-05-16 per DESIGN_DECISIONS.md DD-011. Two recent sessions
opened with handoff notes that claimed verified state but the
verification had not been performed; in both cases entire sessions were
lost to investigating the resulting gap. This protocol exists so that
"tests passing" never substitutes for "smoke test passing, here's the
output."

### 10.1 Canonical end-to-end command

For this project the canonical end-to-end verification is:

```
.\.venv\Scripts\python.exe scripts\evaluate_signals.py --no-report
```

The command must succeed and produce per-signal metrics with `N > 0` and
finite values. The command's runtime should be in the ~5-7 second range
on a warm cache.

When the project grows past the current canonical (e.g. paper trading
adds a "smoke broker connection" step), update this section in the
same commit that introduces the change.

### 10.2 What every handoff must contain

A handoff note that ends a session must include, at minimum:

1. The git commit SHA at HEAD (must match the working tree's HEAD when
   the verification was run).
2. The actual stdout (or final 15-20 lines) of the canonical
   verification command — not a paraphrase, not a claim.
3. The wall-clock runtime.
4. A pass/fail verdict on each signal:
   - **Pass:** per-signal table line (signal name, frequency, at least
     one horizon's IC/ICIR/Sharpe/N).
   - **Fail:** the truncated traceback.

Handoff notes that say "tests passing" or "smoke test runs in ~Xs"
without the actual output are non-compliant. Tests passing does not
imply integration is working. The DD-009 → DD-010 gap is the
canonical example: 203 tests passed for the entire commit window
while the runner had been broken since DD-009 landed.

### 10.3 Cost-benefit

Running the smoke test once at session end is ~6 seconds.
Skipping it has been measured in sub-sessions of investigation. The
discipline is asymmetric in our favor.

### 10.4 Format

A minimal compliant handoff block:

```
Session end — handoff
HEAD: ff92ff4
Canonical verification: scripts\evaluate_signals.py --no-report
Runtime: ~6s
Status: PASS (3/3 signals)

  rates_trend (daily):
    horizon=21  IC=+0.0123  ICIR=+0.1254  Hit=0.5086  Sharpe=+0.1691  N=3551
  fx_carry (monthly):
    horizon=1   IC=-0.0017  ICIR=-0.0031  Hit=0.4991  Sharpe=-0.0870  N=1086
  equity_momentum (monthly):
    horizon=1   IC=+0.0484  ICIR=+0.0870  Hit=0.4900  Sharpe=+0.4425  N=1353

Next session: [free text]
```

If the canonical command fails, paste the traceback instead of the
per-signal table and surface in the next session's opening message.
